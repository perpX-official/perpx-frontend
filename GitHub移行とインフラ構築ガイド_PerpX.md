## エグゼクティブサマリー

本ガイドでは、PerpXプロジェクトのManusからGitHubへの移行、バックエンドインフラの構築、そして大規模ユーザーに対応するためのスケーラビリティ戦略について解説します。

**重要な結論**:
1. **フロントエンド（UI/UX）**: Manusで継続運用（またはVercelへ移行）
2. **スマートコントラクト**: GitHubで管理、Base上にデプロイ
3. **バックエンドサーバー**: 現時点では不要（将来的にAWS/Alibaba Cloudを検討）
4. **スケーラビリティ**: Baseブロックチェーン自体が高スケーラブル

## 1. PerpXのアーキテクチャ全体像

### 1.1. 3層アーキテクチャ

PerpXは、以下の3つのレイヤーで構成されます。

```
┌─────────────────────────────────────────┐
│     Layer 1: フロントエンド (UI/UX)      │
│   - React + RainbowKit                  │
│   - Manus / Vercel でホスティング        │
│   - perpx.fi ドメイン                    │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│  Layer 2: スマートコントラクト (Base)    │
│   - Vault (流動性プール)                 │
│   - PositionManager (ポジション管理)     │
│   - PriceFeed (Chainlink統合)           │
│   - FeeManager (手数料管理)             │

                  │
                  ▼

│   - API（オプション）                    │

└─────────────────────────────────────────┘
```

### 1.2. 各レイヤーの役割

**Layer 1: フロントエンド（UI/UX）**

ユーザーが直接操作するWebアプリケーションです。現在Manusでホスティングされており、perpx.fiドメインで公開されています。

**主な機能**:
- ウォレット接続（RainbowKit）
- 取引画面（TradingViewチャート、注文フォーム）
- ポジション管理画面

- 統計・ダッシュボード

**技術スタック**:
- React 19
- Tailwind CSS 4
- ethers.js（ブロックチェーン接続）
- RainbowKit（ウォレット接続）

**ホスティング**:
- 現在: Manus
- 推奨: Manus継続、またはVercelへ移行

**Layer 2: スマートコントラクト（Base）**

Baseブロックチェーン上にデプロイされるスマートコントラクトです。すべての取引ロジック、ポジション管理、流動性プール管理がここで行われます。

**主な機能**:
- 流動性プールの管理（Vault）
- ポジションの開設/クローズ（PositionManager）
- 価格取得（PriceFeed + Chainlink）

**技術スタック**:
- Solidity 0.8.20+
- Hardhat（開発フレームワーク）
- Chainlink Price Feeds
- OpenZeppelin Contracts

**デプロイ先**:
- Base Mainnet

**Layer 3: バックエンドサーバー（任意）**

オフチェーンでのデータインデックス化やAPIを提供するサーバーです。**現時点では不要**ですが、将来的にユーザー体験向上のために導入を検討できます。

**主な機能（オプション）**:
- 取引履歴のインデックス化（The Graph）
- 統計データのAPI提供
- ユーザー通知（メール、プッシュ通知）

**技術スタック（将来的）**:
- Node.js / Python
- PostgreSQL / MongoDB
- The Graph（ブロックチェーンデータのインデックス化）

## 2. ManusからGitHubへの移行

### 2.1. なぜGitHubが必要か

現在、PerpXのフロントエンド（UI/UX）はManusで管理されていますが、スマートコントラクトの開発には**GitHubでのバージョン管理が必須**です。

**理由**:

1. **チーム開発**: 複数の開発者が同時にスマートコントラクトを開発する場合、GitHubでのバージョン管理が不可欠
2. **監査**: セキュリティ監査会社は、GitHubリポジトリを要求する
3. **透明性**: DeFiプロジェクトでは、スマートコントラクトのソースコードを公開することが一般的
4. **CI/CD**: 自動テスト、自動デプロイなど、CI/CDパイプラインの構築

### 2.2. 移行対象

**GitHubに移行するもの**:
- スマートコントラクト（Solidity）
- テストコード（Hardhat）
- デプロイスクリプト
- ドキュメント

**Manusに残すもの**:
- フロントエンド（React + Tailwind CSS）
- UI/UXデザイン

**または、すべてをGitHubに移行**:
- フロントエンド + スマートコントラクトを1つのモノレポで管理
- Vercelでフロントエンドをホスティング

### 2.3. GitHub移行手順

#### ステップ1: GitHubリポジトリの作成

1. [GitHub](https://github.com/)にアクセスし、新しいリポジトリを作成
2. リポジトリ名: `perpx-contracts`（スマートコントラクトのみ）または `perpx`（フロントエンド + スマートコントラクト）
3. 可視性: Private（初期段階）→ Public（ローンチ後）

#### ステップ2: Manusからコードをエクスポート

Manusプロジェクトのコードをローカルにダウンロードします。

**方法**:
1. Manus Management UIの「Code」パネルから「Download all files」をクリック
2. ZIPファイルをダウンロードして解凍

#### ステップ3: GitHubにプッシュ

ローカルでGitリポジトリを初期化し、GitHubにプッシュします。

```bash
# ダウンロードしたプロジェクトディレクトリに移動
cd perpx

# Gitリポジトリを初期化
git init

# すべてのファイルをステージング
git add .

# 初回コミット
git commit -m "Initial commit: PerpX project from Manus"

# GitHubリポジトリをリモートとして追加
git remote add origin https://github.com/YOUR_USERNAME/perpx.git

# GitHubにプッシュ
git push -u origin main
```

#### ステップ4: プロジェクト構造の整理

GitHubにプッシュした後、プロジェクト構造を整理します。

**推奨構造（モノレポ）**:

```
perpx/
├── frontend/              # フロントエンド
│   ├── src/
│   ├── public/
│   ├── package.json
│   └── vite.config.ts
├── contracts/             # スマートコントラクト
│   ├── src/
│   │   ├── Vault.sol
│   │   ├── PositionManager.sol
│   │   ├── PriceFeed.sol
│   │   ├── FeeManager.sol
│   │   └── Liquidator.sol
│   ├── test/
│   ├── scripts/
│   ├── hardhat.config.js
│   └── package.json
├── docs/                  # ドキュメント
│   ├── architecture.md
│   ├── deployment.md
│   └── security.md
├── .github/               # GitHub Actions
│   └── workflows/
│       ├── test.yml
│       └── deploy.yml
├── README.md
└── package.json
```

**推奨構造（分離）**:

フロントエンドとスマートコントラクトを別々のリポジトリで管理する場合：

- `perpx-frontend`: フロントエンド
- `perpx-contracts`: スマートコントラクト

### 2.4. フロントエンドのホスティング

#### オプション1: Manusで継続（推奨）

現在のManusホスティングを継続します。

**メリット**:
- 現在の設定をそのまま使用可能
- perpx.fiドメインも継続使用
- 追加コストなし（Manusの料金のみ）

**デメリット**:
- Manusプラットフォームへの依存

#### オプション2: Vercelへ移行

GitHubリポジトリをVercelに接続し、自動デプロイを設定します。

**メリット**:
- GitHubとの完全統合
- 自動デプロイ（GitHubにプッシュすると自動的にデプロイ）
- 無料プラン（個人プロジェクト）
- 高速なCDN

**デメリット**:
- 初期設定が必要
- perpx.fiドメインの再設定が必要

**Vercel移行手順**:

1. [Vercel](https://vercel.com/)にアクセスし、GitHubアカウントで登録
2. 「New Project」をクリック
3. GitHubリポジトリ（`perpx`または`perpx-frontend`）を選択
4. フレームワーク: Vite
5. ルートディレクトリ: `frontend`（モノレポの場合）
6. 「Deploy」をクリック
7. デプロイ完了後、perpx.fiドメインを設定

**perpx.fiドメイン設定**:

1. Vercelプロジェクトの「Settings」→「Domains」
2. 「Add Domain」をクリックし、`perpx.fi`を入力
3. CloudflareでCNAMEレコードを追加:
   - Name: `@`（ルートドメイン）または `www`
   - Target: Vercelが提供するCNAME（例：`cname.vercel-dns.com`）
4. DNS設定が反映されるまで待機（数分〜数時間）

## 3. バックエンドインフラの必要性

### 3.1. スマートコントラクト vs バックエンドサーバー

**重要**: GMX方式のPerpetual DEXでは、**バックエンドサーバーは基本的に不要**です。

すべての取引ロジックは**スマートコントラクト（Base上）**で実行されます。フロントエンドは、ethers.jsを使用してスマートコントラクトと直接通信します。

```
フロントエンド (React) 
    ↓ ethers.js
スマートコントラクト (Base)
    ↓ Chainlink
オラクル (価格データ)
```

**バックエンドサーバーが不要な理由**:

1. **取引ロジックはすべてオンチェーン**: ポジション管理、清算、手数料計算など、すべてスマートコントラクトで実行
2. **価格データはChainlinkから取得**: バックエンドサーバーで価格を管理する必要なし
3. **ウォレット接続で認証**: ユーザー認証はウォレット署名で実現、バックエンドでのユーザー管理不要

### 3.2. バックエンドサーバーが必要になる場合

以下の機能を追加する場合、バックエンドサーバーが必要になります。

**1. 取引履歴のインデックス化**

ブロックチェーンから直接取引履歴を取得すると遅いため、オフチェーンでインデックス化します。

**解決策**: The Graph（分散型インデックス化サービス）を使用

**2. 統計データのAPI提供**

TVL、取引量、手数料収入などの統計データをAPIで提供します。

**解決策**: The GraphまたはNode.js APIサーバー

**3. ユーザー通知**

メール通知、プッシュ通知などを提供します。

**解決策**: Node.js + SendGrid（メール）+ Firebase（プッシュ通知）

**4. ソーシャル機能**

リーダーボード、トレーダープロフィール、フォロー機能などを提供します。

**解決策**: Node.js + PostgreSQL

### 3.3. The Graphによるインデックス化（推奨）

The Graphは、ブロックチェーンデータをインデックス化し、GraphQL APIで提供する分散型サービスです。

**メリット**:
- バックエンドサーバー不要
- 分散型（中央集権的なサーバーに依存しない）
- 高速なクエリ

**使用例**:

```graphql
query {
  positions(where: { trader: "0x..." }) {
    id
    size
    collateral
    averagePrice
    isLong
    createdAt
  }
}
```

**The Graph統合手順**:

1. The Graph Studioでサブグラフを作成
2. スマートコントラクトのイベントを定義
3. マッピング関数を実装（TypeScript）
4. サブグラフをデプロイ
5. フロントエンドからGraphQL APIを呼び出し

**コスト**: 無料（開発環境）、$0.0001/クエリ（本番環境）

## 4. スケーラビリティ戦略

### 4.1. ユーザー数の増加に対応する方法

PerpXがユーザー数を増やしても、**Baseブロックチェーン自体が高スケーラブル**なため、インフラ面での心配は最小限です。

**Baseのスケーラビリティ**:

| 項目 | 性能 |
|------|------|
| ブロック時間 | 2秒 |
| TPS（Transactions Per Second） | 1,000+ |
| ガス料金 | $0.10〜$0.30 |
| 最大同時ユーザー | 数百万人 |

**結論**: Baseは、数百万人のユーザーに対応できます。

### 4.2. フロントエンドのスケーラビリティ

フロントエンド（UI/UX）のスケーラビリティは、ホスティングサービスに依存します。

**Manusの場合**:

Manusは、CDN（Content Delivery Network）を使用しているため、大量のユーザーに対応できます。

**Vercelの場合**:

Vercelは、世界中にCDNを展開しており、無制限のスケーラビリティを提供します。

**結論**: フロントエンドのスケーラビリティは問題ありません。

### 4.3. スマートコントラクトのスケーラビリティ

スマートコントラクトは、Baseブロックチェーン上で実行されるため、Baseのスケーラビリティに依存します。

**ガス最適化**:

スマートコントラクトのガス消費を最適化することで、ユーザーの取引コストを削減できます。

**最適化手法**:
- `uint256`の代わりに`uint128`を使用（必要に応じて）
- 不要なストレージ書き込みを削減
- `view`関数を活用
- イベントの効率的な使用

### 4.4. バックエンドサーバーのスケーラビリティ（将来的）

将来的にバックエンドサーバーを導入する場合、以下のスケーラビリティ戦略を採用します。

**1. The Graphの使用**

The Graphは、分散型でスケーラブルなインデックス化サービスです。

**2. AWS / Alibaba Cloudのオートスケーリング**

ユーザー数に応じて、サーバーインスタンスを自動的に増減します。

**3. CDNの活用**

静的コンテンツ（画像、CSS、JavaScriptなど）をCDNでキャッシュします。

**4. データベースのレプリケーション**

読み取り負荷を分散するため、データベースのレプリカを作成します。

## 5. Alibaba Cloud / AWS統合戦略

### 5.1. 現時点では不要

**重要**: 現時点では、Alibaba CloudやAWSは不要です。

**理由**:

1. **スマートコントラクトはBase上で実行**: サーバーレス
2. **フロントエンドはManus/Vercelでホスティング**: サーバーレス
3. **バックエンドサーバーは不要**: すべてオンチェーンで完結

### 5.2. 将来的に必要になる場合

以下の機能を追加する場合、Alibaba CloudまたはAWSが必要になります。

**1. カスタムAPIサーバー**

The Graphでは実現できない複雑なロジックをAPIで提供する場合。

**2. ユーザー通知サービス**

メール通知、プッシュ通知などを提供する場合。

**3. ソーシャル機能**

リーダーボード、トレーダープロフィール、フォロー機能などを提供する場合。

**4. 高度な分析・レポート**

ユーザー行動分析、トレード分析などを提供する場合。

### 5.3. AWS vs Alibaba Cloud

将来的にクラウドサーバーが必要になった場合、AWSとAlibaba Cloudのどちらを選ぶべきでしょうか。

**AWS推奨**:

| 項目 | AWS | Alibaba Cloud |
|------|-----|---------------|
| DeFi/ブロックチェーンエコシステム | ★★★★★ | ★★★☆☆ |
| グローバルカバレッジ | ★★★★★ | ★★★★☆ |
| 日本語サポート | ★★★★☆ | ★★★★★ |
| コスト（アジア） | ★★★☆☆ | ★★★★★ |
| ドキュメント | ★★★★★ | ★★★★☆ |

**推奨**: **AWS**（DeFi/ブロックチェーンエコシステムが最も成熟）

ただし、アジア市場重視でコスト削減を優先する場合は、Alibaba Cloudも有力な選択肢です。

## 6. ドメイン・DNS管理

### 6.1. 現在の構成

現在のperpx.fiドメインの構成は以下の通りです。

**ドメイン管理**: Netim  
**DNS管理**: Cloudflare  
**ホスティング**: Manus  
**メール**: ZohoMail

### 6.2. DNS設定の確認

Cloudflareで以下のDNSレコードが設定されているはずです。

| Type | Name | Target | 説明 |
|------|------|--------|------|
| CNAME | @ または www | Manusが提供するCNAME | Webサイト |
| MX | @ | ZohoMailのMXレコード | メール |
| TXT | @ | SPF, DKIM, DMARC | メール認証 |

### 6.3. Vercelへ移行する場合のDNS設定

Vercelへ移行する場合、CloudflareのCNAMEレコードを変更します。

**変更前**:
```
Type: CNAME
Name: @
Target: [Manus CNAME]
```

**変更後**:
```
Type: CNAME
Name: @
Target: cname.vercel-dns.com
```

**注意**: MXレコード（メール）は変更しないでください。

### 6.4. Netimの役割

Netimはドメインレジストラとしてのみ機能しています。DNS設定はCloudflareで管理されているため、Netimでの設定変更は不要です。

**Netimで確認すべきこと**:
- ドメインの有効期限
- ネームサーバーがCloudflareに設定されているか

## 8. まとめ

### 8.1. 重要なポイント

**1. バックエンドサーバーは現時点では不要**

GMX方式のPerpetual DEXでは、すべての取引ロジックがスマートコントラクト（Base上）で実行されるため、バックエンドサーバーは不要です。
