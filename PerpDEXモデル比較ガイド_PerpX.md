## エグゼクティブサマリー

Perpetual DEX（無期限先物分散型取引所）には、主に**AMM型**、**Order Book型**、**Hybrid型**、**Oracle型**の4つのモデルが存在します。本ガイドでは、各モデルの仕組み、利点、課題、実装の複雑さを詳細に比較し、PerpXプロジェクトに最適なアーキテクチャを推奨します。

**結論**: PerpXには**Oracle型（GMX方式）**を最推奨します。開発コストが最も低く、迅速なローンチが可能で、流動性要件も最小限です。

---

## 1. Perpetual DEXモデルの概要

Perpetual DEX（Perp DEX）は、有効期限のない無期限先物契約を取引できる分散型取引所です。従来の中央集権型取引所（CEX）と異なり、ブロックチェーン上のスマートコントラクトによって自律的に運営され、ユーザーは秘密鍵を自分で管理します[1]。

2023年3月には、DEXとCEXのスポット取引量の比率が10%を超え、2025年現在もその成長は続いています[2]。しかし、無期限先物市場においては、オンチェーンでの取引はまだ初期段階にあり、大きな成長余地があります。

## 2. 4つのモデルの詳細比較

### 2.1. AMM型（Automated Market Maker）

#### 仕組み

AMM型Perpetual DEXは、**仮想AMM（vAMM）**を使用して価格発見を行います[3]。実際のスポット資産の交換ではなく、名目上の先物契約がAMMで取引されたと仮定し、決済時に残高のみを精算します。

価格は**x × y = K**という方程式に基づいて自動的に決定されます。Uniswapのようなスポット用AMMは実際のトークンで運用されますが、先物用AMMでは仮想的な取引のみが行われます[3]。

#### 主な特徴

**利点**:

AMM型の最大の利点は、**誰でも簡単にマーケットメイカーとして参加できる**ことです。x × y = Kという方程式に従って自動的に価格が決定されるため、プロフェッショナルなトレーダーでなくても、一般ユーザーが流動性提供者として参加できます[3]。

また、**新しい取引ペア追加のコストが最も低い**という利点もあります。Order Book型では新しい資産を追加する際にプロフェッショナルなマーケットメイカーの確保が必要ですが、AMM型ではその制約がありません[3]。

さらに、AMMは**完全にオンチェーンで価格発見が可能**な唯一のモデルです（理論上）。ブロックチェーンのスケーラビリティ制限により、現状では完全に分散化されたオーダーブックを実装することは困難ですが、AMMはプラットフォーム上で生成された取引に基づいて独自に価格を発見できます[3]。

**課題**:

現在のAMM型Perp DEXは取引量が少ないため、**Mark Price（AMM上の取引価格）がIndex Price（市場価格）に追従する必要があります**[3]。これには以下の2つの大きな課題があります。

**課題1: Mark Price vs Index Price**

現在の価格フローは以下のようになっています[3]：

```
CEX先物 → CEXスポット → Oracle → AMM Perp DEX Index Price <> Mark Price
```

Mark PriceとIndex Priceに差がある場合、資金調達率（Funding Rate）を設定し、アービトラージャーを誘導してMark PriceがIndex Priceに追従するようにします。しかし、これには**常にコストが発生**します。

AMM型では、**AMM自体がHouse（ディーラー）として機能**し、すべてのトレーダーとアービトラージャーのポジションのカウンターパーティとなります[3]。これはOrder Book型（ブローカーモデル）とは根本的に異なります。Order Book型では、トレーダー同士が反対のポジションを取り合い、取引所は仲介役に徹します。

AMM型では、オープンインタレストの不均衡による資金調達コスト/利益はプロトコルに帰属します。Index Priceに追従する必要があるAMMでは、資金調達はアービトラージャーへのインセンティブとして決定され、**常にコスト**となります[3]。

**課題2: 仮想K値の設定**

スポット用AMMでは、実際のスポットプールが保有され、取引が行われるため、正確なK値（流動性深度）が設定されます。トレーダーとアービトラージャーが実際のK値に応じて自動的に適切な価格を見つけます[3]。

しかし、先物用AMMでは、実際の資産が移動せず、名目上の仮想取引のみが想定されるため、契約終了時に残高のみが決済されます。そのため、**仮想深度（仮想K）の設定はプロトコルの責任**となります[3]。

- **K値が大きすぎる場合**: 深度が過度に深くなり、価格が本来動くべきほど動かなくなります。アービトラージャーが離脱し、プロトコルが大きな絶対値の資金調達コストを支払う必要があります。
- **K値が小さすぎる場合**: スリッページが高くなり、ユーザーが離脱します。

現在は、実際の流動性を追加して仮想AMMを運用する方法が使用されています。資本効率を犠牲にする必要がありますが、外部から実際の流動性を受け入れることでHouseリスクを共有し、実際のK値を簡単に導出できます[3]。

#### 代表的なプロジェクト

| プロジェクト | 特徴 |
|--------------|------|
| **Perpetual Protocol** | vAMMの先駆者、Optimismベース |
| **Drift Protocol** | Solanaベース、高速取引 |

### 2.2. Order Book型

#### 仕組み

Order Book型Perpetual DEXは、従来の中央集権型取引所（CEX）と同様の仕組みを分散型フレームワークで実装したモデルです[2]。買い注文と売り注文がトレーダーによって発注され、価格と時間優先順位に基づいてマッチングされます。

オーダーブックにはすべてのアクティブな買い注文と売り注文が記録され、マッチングエンジンが注文価格が一致したときに取引を実行します[2]。

#### 主な特徴

**利点**:

Order Book型は、**リテールおよび機関投資家の両方に好まれる取引モデル**です[2]。マーケットメイカーや流動性提供者は、アクティブなマーケットメイキングまたはヘッジ戦略に従うため、これらのPerpetual DEXは**プロフェッショナルなマーケットメイカーによって整えられた豊富な流動性**を誇ります。

また、**低スリッページ**も大きな利点です。正確な注文マッチングにより、スリッページが最小限に抑えられます[2]。

さらに、**参入障壁が低い**という特徴もあります。CEXに似ているため、すべてのクラスのトレーダーが採用・操作しやすいです[2]。

価格発見は、**オーダーブック内の最高買い注文と最低売り注文**によって決定されます[2]。

**課題**:

完全にオンチェーンのPerpetual DEXは、**スループット制限、ブロックチェーンレイテンシ、フロントランニング、操作脆弱性**などの重大な課題に直面します[2]。ただし、SolanaやMonadなどは、インフラレベルでこの問題を解決しようと努力しています。

また、**マーケットメイカー依存**も課題です。新しい資産を追加する際には、プロフェッショナルなマーケットメイカーの存在が必要です[3]。

現状では、完全な分散化のために**オフチェーンオーダーブック**を使用する必要があります[2]。

#### アーキテクチャ

Order Book型は**ブローカーモデル**です。トレーダー同士が反対のポジションを取り合い、取引所は独自のマーケットメイキング以外は仲介役のみを担います。AMM型のようにHouseとして機能しません[3]。

#### 代表的なプロジェクト

| プロジェクト | 特徴 |
|--------------|------|
| **Hyperliquid** | 独自L1チェーン、完全オンチェーンオーダーブック |
| **Zeta** | Solanaベース、オプション取引も提供 |
| **LogX** | マルチチェーン対応 |

#### 実装の複雑さとコスト

| 項目 | 評価 |
|------|------|
| スマートコントラクト開発 | ★★★★★（非常に複雑） |
| 流動性要件 | ★★★★★（マーケットメイカー必須） |
| 初期開発コスト | $300K〜$800K |
| 開発期間 | 10〜18ヶ月 |

### 2.3. Hybrid型（オフチェーンオーダーブック + オンチェーン決済）

#### 仕組み

Hybrid型は、**取引マッチングはオフチェーンで行い、トランザクション実行とそれ以外のプロセスはオンチェーンで実行する**アプローチです[2]。

このハイブリッドアプローチにより、DeFiのカストディ、透明性、セキュリティのメリットを確保しながら、オフチェーンマッチングによってプロセスの速度とパフォーマンスを向上させ、MEV（Maximal Extractable Value）に関連するリスクを軽減します[2]。

#### 主な特徴

**利点**:

Hybrid型は、**DeFiとCEXの両方のメリット**を享受できます。カストディ、透明性、セキュリティを確保しながら、CEXレベルの速度を実現します[2]。

オフチェーンマッチングにより、**MEVリスクが軽減**されます[2]。

また、**低スリッページ**も実現できます。高頻度マーケットメイカーによる深い流動性プールが維持されます[2]。

価格発見は、**CEXと同様の迅速な価格更新と注文マッチング**が可能です。オフチェーン環境により、CEXと同様の価格発見が実現されます[2]。

**課題**:

Hybrid型は、**オフチェーンインフラの構築が必要**です。オーダーブックとマッチングエンジンをオフチェーンで運用するため、複雑なインフラが必要です。

また、**マーケットメイカー依存**も課題です。Order Book型と同様に、プロフェッショナルなマーケットメイカーの存在が必要です[2]。

#### アーキテクチャ

**オフチェーン**:
- オーダーブック
- マッチングエンジン
- 価格更新

**オンチェーン**:
- 決済
- カストディ
- 最終的なトランザクション実行

### 2.4. Oracle型

#### 仕組み

Oracle型Perpetual DEXは、**外部オラクル（Chainlinkなど）から市場価格を取得し、独自の価格発見メカニズムを持たない**モデルです。

トレーダーは、オラクルが提供する価格で直接取引を行います。流動性は、ユーザーが預け入れた資産で構成される**流動性プール（Liquidity Pool）**によって提供されます。

#### 主な特徴

**利点**:

Oracle型の最大の利点は、**シンプルな実装**です。価格発見メカニズムが不要なため、開発が大幅に簡素化されます。

また、**低い流動性要件**も大きなメリットです。外部価格に依存するため、独自の深い流動性を確保する必要がありません。マーケットメイカーも不要です。

さらに、**迅速なローンチ**が可能です。開発期間が短く、コストも最小限に抑えられます。

**課題**:

Oracle型の最大の課題は、**オラクル依存**です。外部データソースへの依存により、オラクル攻撃のリスクがあります。

また、**限定的な資産**も課題です。十分な取引量と市場深度がある資産のみが取引可能です[3]。

さらに、**流動性提供者のリスク**もあります。トレーダーの利益は流動性プールの損失となるため、流動性提供者は慎重にリスク管理を行う必要があります。

#### アーキテクチャ

**主要コンポーネント**:
- **流動性プール（GLP、gDAIなど）**: ユーザーが預け入れた資産で構成
- **オラクル（Chainlink Price Feeds）**: 外部から市場価格を取得
- **スマートコントラクト**: ポジション管理、決済、手数料計算

#### 代表的なプロジェクト

| プロジェクト | 特徴 |
|--------------|------|
| **GMX** | Arbitrum/Avalanche、GLP流動性プール |
| **Gains Network** | Polygon、gDAI流動性プール |

#### 実装の複雑さとコスト

| 項目 | 評価 |
|------|------|
| スマートコントラクト開発 | ★★★☆☆（中程度） |
| 流動性要件 | ★★★☆☆（中程度、マーケットメイカー不要） |
| 初期開発コスト | $85K〜$180K |
| 開発期間 | 5〜8ヶ月 |

### 3.2. 各モデルの適性

#### AMM型が適している場合
- DeFiネイティブなユーザーをターゲットにする
- 完全なオンチェーン分散化を重視する
- 十分な初期流動性を確保できる
- 長期的な成長を見据えている

#### Order Book型が適している場合
- CEXユーザーを取り込みたい
- 機関投資家をターゲットにする
- マーケットメイカーとの提携が可能
- 高い開発予算がある

#### Hybrid型が適している場合
- CEXレベルの速度とDeFiのメリットを両立したい
- 大規模な取引量を想定している
- オフチェーンインフラを構築できる
- 最高レベルの開発予算がある

#### Oracle型が適している場合
- **迅速なローンチを優先する**
- **開発コストを最小限に抑えたい**
- **マーケットメイカーを確保できない**
- **初期段階のプロジェクト**

## 4. PerpXへの推奨

### 4.1. 推奨モデル: **Oracle型（GMX方式）**

PerpXプロジェクトには、**Oracle型（GMX方式）**を最推奨します。

#### 推奨理由

**1. 最も低い開発コスト**

Oracle型の初期開発コストは**$85K〜$180K**で、他のモデルと比較して圧倒的に低コストです。

| モデル | 開発コスト |
|--------|------------|
| Oracle型 | $85K〜$180K |
| AMM型 | $200K〜$500K |
| Order Book型 | $300K〜$800K |
| Hybrid型 | $400K〜$1M+ |

**2. 最も短い開発期間**

開発期間は**5〜8ヶ月**で、最も迅速にローンチできます。

| モデル | 開発期間 |
|--------|----------|
| Oracle型 | 5〜8ヶ月 |
| AMM型 | 8〜12ヶ月 |
| Order Book型 | 10〜18ヶ月 |
| Hybrid型 | 12〜24ヶ月 |

**3. マーケットメイカー不要**

Oracle型は、流動性プールによって流動性が提供されるため、**プロフェッショナルなマーケットメイカーを確保する必要がありません**。これは、初期段階のプロジェクトにとって大きなメリットです。

**4. 実績のあるモデル**

GMXは、Oracle型の代表的なプロジェクトとして、**$10億以上のTVL**を達成しています。実績のあるモデルを採用することで、リスクを最小限に抑えられます。

**5. Base上での実装が容易**

Baseは完全にEVM互換なので、GMXのスマートコントラクトをベースに、Baseに最適化した実装が可能です。

### 4.2. 実装アーキテクチャ

#### 主要コンポーネント

**1. 流動性プール（PLP - PerpX Liquidity Pool）**

ユーザーが複数の資産（ETH、USDC、WBTCなど）を預け入れるプールです。トレーダーの取引のカウンターパーティとして機能します。

**2. オラクル（Chainlink Price Feeds）**

Chainlinkから市場価格を取得します。Baseは[Chainlinkをサポート](https://docs.chain.link/data-feeds/price-feeds/addresses?network=base)しているため、すぐに統合できます。

**3. ポジション管理スマートコントラクト**

ポジションの開設、クローズ、清算を管理します。

**4. 手数料・報酬システム**

取引手数料、資金調達率、流動性提供者への報酬を管理します。

#### 技術スタック

| レイヤー | 技術 |
|----------|------|
| ブロックチェーン | **Base** |
| スマートコントラクト | Solidity 0.8.x |
| オラクル | Chainlink Price Feeds |
| フロントエンド | React + ethers.js |
| 開発フレームワーク | Hardhat |

- 流動性プール設計

- Base Mainnetへのデプロイ
- 初期流動性の確保
- マーケティング・コミュニティ構築

**総期間**: **5〜8ヶ月**

### 4.4. 初期流動性の確保

Oracle型では、初期流動性の確保が重要です。以下の方法を推奨します。

**1. チームによる初期流動性提供**

プロジェクトチームが初期流動性（$100K〜$500K相当）を提供します。

**2. 流動性マイニングプログラム**

流動性提供者に対して、PerpXトークンを報酬として配布します。

**3. VC・投資家からの流動性**

VCや投資家に流動性提供を依頼します。

**4. パートナーシップ**

他のDeFiプロトコルとのパートナーシップにより、流動性を確保します。

### 4.5. リスクと対策

#### リスク1: オラクル攻撃

**対策**:
- Chainlinkのような信頼性の高いオラクルを使用
- 複数のオラクルソースを統合（Chainlink + Pyth Network）
- 異常な価格変動を検知するサーキットブレーカーを実装

#### リスク2: 流動性プールの枯渇

**対策**:
- 十分な初期流動性を確保
- 流動性提供者への魅力的な報酬
- 最大レバレッジの制限

#### リスク3: スマートコントラクトの脆弱性

**対策**:
- 複数のセキュリティ監査会社による監査
- バグバウンティプログラムの実施
- 段階的なローンチ（小規模→大規模）

## 6. まとめ

Perpetual DEXには、AMM型、Order Book型、Hybrid型、Oracle型の4つのモデルが存在します。それぞれに利点と課題がありますが、PerpXプロジェクトには**Oracle型（GMX方式）**を最推奨します。

Oracle型は、**最も低い開発コスト（$85K〜$180K）**、**最も短い開発期間（5〜8ヶ月）**、**マーケットメイカー不要**という特徴を持ち、初期段階のプロジェクトに最適です。

Baseブロックチェーン上でOracle型Perpetual DEXを構築することで、低コスト、高速、Coinbaseエコシステムへのアクセスという3つのメリットを享受できます。
